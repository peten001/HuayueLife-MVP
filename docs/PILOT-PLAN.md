# 《华越优选 V1.0 Alpha 试点落地计划》

## 1. 当前 MVP 功能清单

### 用户端

- 微信登录及 JWT 登录状态
- 北宁、北江城市选择
- 定位附近商家
- 拒绝定位后按城市展示
- 商家详情和菜单
- 扫描桌台二维码
- 堂食桌号绑定
- 购物车
- 订单预览和价格复核
- 堂食、到店自取、商家配送下单
- 幂等提交，防止重复订单
- 用户订单列表和详情
- 待接单订单取消
- 配送订单确认收货
- 订单状态轮询

### 商家后台

- 商家员工登录
- 商家资料和营业时间设置
- 堂食、自取、商家配送开关
- 配送费、起送价、配送半径设置
- 分类和菜品管理
- 菜品上架、售罄、下架
- 桌台管理
- 二维码生成、下载和换码
- 今日经营看板
- 订单列表和详情
- 接单、拒单、制作、完成、配送状态操作
- 新订单声音提醒
- 标记已收款
- 商家间数据隔离

### 明确不包含

- 在线支付
- 骑手系统
- 优惠券
- 会员积分
- 菜品规格和加料
- 厨房打印机
- 多行业商城

## 2. 上线阻断项

### 必须完成

- 填写真实微信 AppID 和 AppSecret
- 使用体验版验证真实 `code2Session`
- 部署 HTTPS API 域名
- 配置微信 request 合法域名
- 完成微信隐私保护指引
- 替换 JWT、数据库及试点账号密码
- 执行生产数据库迁移
- 验证数据库备份和恢复
- 使用真实手机完成三种订单流程
- 替换所有试点示例资料和二维码

### 当前发布定位

`MVP-V1.0-alpha` 适合受控试点，不应在未完成上述事项前全面公开推广。

## 3. 微信小程序准备事项

1. 在微信公众平台取得正式 AppID。
2. 将 AppID 写入小程序 `manifest.json`。
3. 在 API 生产环境配置：

   ```env
   WECHAT_APP_ID="真实AppID"
   WECHAT_APP_SECRET="真实AppSecret"
   ```

4. 配置 request 合法域名：

   ```text
   https://api.example.com
   ```

5. 完成隐私保护指引，声明：
   - 微信用户标识
   - 联系电话
   - 配送地址
   - 精确位置
6. 使用生产 API 地址构建：

   ```bash
   VITE_API_BASE_URL="https://api.example.com/api/v1" \
   corepack pnpm --filter @huayue-life/miniapp build
   ```

7. 导入微信开发者工具：

   ```text
   apps/miniapp/dist/build/mp-weixin
   ```

8. 上传体验版并配置试点测试人员。
9. 检查 TabBar：首页、订单、我的。
10. 验证拒绝定位后的城市降级流程。

## 4. 服务器准备事项

### 推荐资源

试点期单台应用服务器即可：

- 2 CPU
- 4 GB 内存
- 40 GB SSD
- Linux LTS
- MySQL 8
- Nginx
- Node.js 兼容版本
- Redis 7 可预留，但当前不是核心依赖

### 部署要求

- API 和商家后台启用 HTTPS
- MySQL、Redis 不开放公网端口
- 使用独立数据库应用账号
- 配置生产 `.env`
- 执行：

  ```bash
  corepack pnpm install --frozen-lockfile
  corepack pnpm typecheck
  corepack pnpm build
  corepack pnpm --filter @huayue-life/api prisma:deploy
  corepack pnpm --filter @huayue-life/api start:prod
  ```

- 配置进程自动重启
- 配置日志保存和磁盘轮转
- 配置每日数据库备份
- 检查：

  ```text
  GET https://api.example.com/api/v1/health
  ```

## 5. 第一家试点餐厅导入步骤

1. 确认店主和技术联系人。
2. 收集商家资料：
   - 中越文名称
   - Logo、封面
   - 地址、经纬度
   - 联系电话
   - 营业时间
3. 确认支持的订单类型。
4. 确认配送费、起送价和配送范围。
5. 收集菜单：
   - 分类
   - 菜名
   - 图片
   - 价格
   - 描述
6. 导入或后台录入真实菜单。
7. 商家逐项核对名称和价格。
8. 建立真实桌号。
9. 创建店主账号并使用独立强密码。
10. 在后台逐桌执行换码。
11. 下载并打印二维码。
12. 培训商家完成一次完整接单演练。
13. 试营业前冻结菜单修改，进行最终核价。

## 6. 二维码打印方案

### 二维码内容

- 使用随机 `qr_token`
- 不包含数据库桌台 ID
- 每张二维码只对应一个商家和桌号

### 建议版式

二维码卡片包含：

- 餐厅名称
- “微信扫码点餐”
- 桌号，例如“A01”
- 二维码
- “如无法下单请联系服务员”
- 中越双语提示

### 打印规格

- 建议尺寸：10 cm × 15 cm
- 二维码有效区域不小于 4 cm × 4 cm
- 使用哑光覆膜，减少灯光反射
- 深色二维码配白色背景
- 不裁切二维码留白区域
- 每张卡片背面标注内部桌号

### 现场验收

- 每桌至少使用两台不同手机扫码
- 核对显示的商家名称和桌号
- 检查远距离、弱光和倾斜角度识别
- 换码后验证旧码立即失效
- 保留一套备用二维码卡片

## 7. 试营业测试流程

### 堂食

1. 扫描桌面二维码。
2. 核对商家和桌号。
3. 添加菜品并提交订单。
4. 检查商家后台声音提醒。
5. 商家接单并开始制作。
6. 标记制作完成。
7. 商家完成整单。
8. 标记已收款。
9. 用户订单页确认状态同步。

### 到店自取

1. 从附近商家进入菜单。
2. 选择到店自取。
3. 填写联系人和电话。
4. 提交订单。
5. 商家完成接单、制作、取餐和收款流程。

### 商家配送

1. 选择商家配送。
2. 填写联系人、电话和地址。
3. 测试定位范围校验及手填地址。
4. 商家接单并制作。
5. 商家开始配送。
6. 用户确认收货。
7. 检查订单变为已完成。

### 异常测试

- 无效二维码
- 换码后的旧二维码
- 停用桌台
- 菜品售罄或下架
- 下单时价格变化
- 重复点击提交
- 商家接单后用户取消
- 非配送订单确认收货
- 用户拒绝定位
- 网络中断后重试
- 跨用户、跨商家访问订单

## 8. 第一周运营观察指标

### 订单指标

- 每日订单数
- 堂食、自取、配送订单占比
- 下单成功率
- 重复订单数量
- 取消和拒单率
- 订单完成率

### 时效指标

- 下单到接单平均时间
- 接单到开始制作时间
- 制作平均时间
- 配送平均时间
- 待接单超过 3 分钟的订单数

### 系统指标

- API 错误率
- 微信登录失败率
- 二维码解析失败率
- 订单创建失败率
- 数据库和服务器资源使用率
- 前端反馈的空白页或卡顿次数

### 商家运营指标

- 声音提醒漏听次数
- 菜品售罄更新是否及时
- 错桌、漏单和重复制作次数
- 价格错误次数
- 已完成但未标记收款的订单数

每日营业结束后由店主和项目负责人进行一次订单核对。

## 9. V1.1 版本规划

V1.1 应根据试点数据排期，不提前扩展行业。

建议优先级：

1. 微信用户资料和手机号完善
2. 商家员工密码修改和账号管理
3. 订单超时提醒
4. 更可靠的订单实时通知，评估 SSE 或 WebSocket
5. 商家订单打印或厨房小票接入
6. 菜品批量导入和图片上传
7. 营业时间跨午夜支持优化
8. 订单搜索和数据导出
9. 用户常用联系人及配送地址
10. 监控、告警、限流和审计日志

暂不进入 V1.1：

- 在线支付
- 平台结算
- 骑手系统
- 优惠券和积分
- 水果、鲜花、蛋糕等多行业功能

## 10. 风险清单

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 微信审核或隐私声明不通过 | 无法发布正式版 | 提前使用体验版核对权限和隐私声明 |
| code2Session 配置错误 | 用户无法登录 | 核对 AppID/AppSecret，监控登录失败率 |
| HTTPS 或合法域名错误 | 小程序无法请求 API | 发布前使用开启域名校验的真机测试 |
| 默认密码未替换 | 商家后台被未授权访问 | Seed 使用独立强密码，安全渠道交付 |
| 二维码贴错桌位 | 订单绑定错误桌号 | 打印、张贴、扫码三次核对 |
| 商家漏听新订单 | 延迟接单 | 开启声音、保持页面前台、设置人工巡检 |
| 网络不稳定 | 下单或状态更新延迟 | 幂等提交、明确错误提示、允许刷新重试 |
| 菜品价格错误 | 经营损失和投诉 | 商家上线前逐项核价 |
| 配送地址不准确 | 无法送达 | 手填地址配合电话确认 |
| 数据库故障 | 订单数据不可用 | 每日备份、迁移前备份、恢复演练 |
| 单机故障 | 试点服务中断 | 进程自动重启、健康检查和故障联系人 |
| 日志泄露敏感信息 | 用户或密钥风险 | 禁止记录 code、AppSecret、session_key |
| 状态操作错误 | 订单流程异常 | 服务端状态机校验和状态日志审计 |
| 试点需求快速膨胀 | 上线延期 | 冻结 V1.0 范围，新需求进入 V1.1 评审 |
| 当前无后台改密页面 | 密码管理不便 | 部署时设置强密码，V1.1 增加改密能力 |

**试点放行标准：** 所有上线阻断项完成，三种订单类型真机流程通过，试点商家完成培训，数据库备份恢复验证成功。
